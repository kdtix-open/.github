# Contributing to KDTIX Open‑Source Projects

Thank you for taking the time to contribute!  We welcome contributions in many
forms: filing bug reports, suggesting features, improving documentation,
triaging issues or submitting code changes.  The guidelines below are
adapted from proven open‑source projects such as `awesome` and
`freeCodeCamp`.  They help us keep our maintainers’ workload manageable and
our community healthy.

## Before You Begin

* **Search existing issues and pull requests.**  If you have a question or
  find a bug, check the issue tracker first.  Avoid opening duplicate
  reports.  Closed issues may also contain useful
  context or explanations.
* **Be respectful.**  Please read our [Code of Conduct](CODE_OF_CONDUCT.md) and
  treat everyone with kindness.  We expect
  contributors to foster a welcoming, collaborative environment.
* **Decide whether you need an issue.**  For small fixes (typos, simple
  documentation improvements) you can submit a pull request directly.  For
  larger changes or new features, open an issue first to discuss your
  proposal.

## How to Contribute

1. **Fork** the repository you want to improve and clone it locally.
2. **Create a topic branch** (`git checkout -b my-feature`) off of the default
   branch.  Avoid working directly on `main` or `master`.
3. **Make your changes**, adhering to the existing code style.  Add tests if
   relevant and update documentation.  Avoid unrelated changes.
4. **Squash your commits** into a single coherent commit before submitting
   your pull request, unless there is a good reason to keep them separate.
5. **Push** your branch to your fork and open a **pull request** against the
   `kdtix-open` repository.  Give the pull request a descriptive title and
   explain what it does and why.  If your change
   addresses an issue, mention it in the description using `Fixes #123`.
6. **Check the boxes** in the pull request template to confirm you have read
   our guidelines and completed the required steps.
7. **Address review comments.**  Maintainers may ask questions or request
   changes.  Respond politely and update your pull request accordingly.

## Pull Request Tips

* Keep your pull request focused on a single improvement and avoid mixing
  unrelated changes.
* Review the diff of your changes before submitting.  It helps catch
  typos and formatting issues.
* Write commit messages in the imperative mood (“Add support for X”), not in
  the past tense.  Use present tense for the pull request title as well.
* Ensure the “Allow edits from maintainers” checkbox is enabled to let
  maintainers make minor changes.
* Be patient.  Our team reviews pull requests as time allows.  Feel free to
  leave a friendly reminder if you haven’t heard back after a couple of
  weeks.

## Bug Reports

When filing a bug report, please include as much detail as possible.  A
minimal reproduction (for example, a script or a set of steps that trigger the
problem) helps us diagnose the issue quickly.  If you
know how to fix the bug, consider sending a pull request with a failing test
and the proposed fix.

## Code of Conduct

By participating in this project, you agree to abide by our
[Contributor Covenant](CODE_OF_CONDUCT.md).  Instances of abusive or
harassing behaviour will not be tolerated, and may lead to a temporary or
permanent ban.
